# 专业模式文案修改建议：Apple 风格

## 依据

- Memory 里只有项目偏好：面向非技术用户、iOS 26 / Liquid Glass 气质、避免抽象文案、默认 WCAG AA 和直接明白的语言。
- Apple 官方资料已补查：
  - Apple Style Guide June 2025：适用于 instructional materials、technical documentation、reference information、training programs 和 user interfaces，目标是保持一致 voice；第三方开发者也应遵循 user-facing text 指南。
  - Apple Human Interface Guidelines / Writing：应用内文字是用户体验的一部分。
  - Apple Human Interface Guidelines / Inclusion：内容和功能要让所有人都能访问和理解，沟通要尊重、以人为先。
  - Apple Human Interface Guidelines / Accessibility：可访问界面让每个人都能获得良好体验。

## 目标

专业模式不是说明书页面，而是一个安静、高效的生图 App。文案要让用户立刻知道下一步做什么，减少工程词、教学段落和自夸式营销。

## 总原则

1. 使用简体中文，避免繁中、乱码、英文 UI 词；文件类型可写作 `ZIP 文件`、`JSON 配置`，但普通用户入口尽量隐藏技术名。
2. 控件名 2-6 字；说明文案 1 句内，最长不超过 24 个汉字。
3. 先说用户结果，再说系统机制。不要把“AI / 系统 / 工作流”当主角。
4. 保留专业能力，但降低术语门槛：`反向提示词` 改为 `避免内容`，`项目上下文` 改为 `项目信息`。
5. 错误和空状态必须给下一步，不只描述状态。
6. 弱化宣传词：禁用 `开箱即用`、`一键就绪`、`完整参数`、`生成溯源`、`架构校验`、`自动包裹提示词外层`。
7. 不在主界面解释实现细节；需要解释的内容放到帮助文档或悬浮帮助。
8. 按 Apple Style Guide 的方向，避免第一人称，避免“允许/使得/让你能够”这类绕句，改成直接动作。

## 场景模板文案规则

- 模板名称保留行业语义，但缩短到 4-8 字：`汽车广告 KV` 可保留，`微信公众号头图` 可改 `公众号头图`。
- `contextPrompt` 不要写成美术总监长 brief；控制在 2 句内：目标 + 画面限制。
- `negativePrompt` 只列具体可避免项，不写抽象评价：用 `文字乱码、低清晰度、肢体畸形`，少用 `廉价感`。
- 场景卡片 `note` 只说结果：`适合电商主图`、`适合开屏海报`、`适合横版广告`。
